# CLAUDE.md

用于减少 LLM 常见编码错误的行为准则。可按需与项目专属说明合并。

**取舍：** 这套规则更偏向"谨慎优先"，而不是"速度优先"。对于非常简单的任务，请自行判断，不必机械套用。

## 项目规则加载

当你在本仓库中工作时，在开始分析、编辑、测试、评审或提交准备之前，必须先读取：

1. `.agent-rules/README.md`（规则读取链路总纲）
2. `.agent-rules/rules/README.md` 索引，按任务定位相关规则文件再精读

读取顺序、目录分层（`rules/`、`mechanisms/`、`dev-plans/`、`handoffs/`）、以及任务相关项目文档的读取时机，统一以 `.agent-rules/README.md` 为准。

附加约束：

- `.agent-rules/` 是本仓库项目级 Agent 规则、机制梳理、开发计划和接力摘要的真来源。
- 新的项目级 Agent 规则添加至 `.agent-rules/` 对应分层目录。
- `CLAUDE.md` 只保留通用行为准则和规则入口摘要，不在这里重复维护整套项目规范。
- 若桥接文件与根 `.agent-rules/` 冲突，以根 `.agent-rules/` 为准。
- 规则优先级为：系统/宿主工具指令 > 用户要求 > 根 `.agent-rules/rules/` > 根 `.agent-rules/mechanisms/` > 本文件或其他 Agent 桥接文件 > 其他项目文档。

## 1. 写代码前先思考

**不要擅自假设。不要掩盖困惑。主动说出取舍。**

在开始实现之前：
- 明确说出你的假设；如果不确定，就先问。
- 如果需求存在多种解释，先列出来，不要默默替用户做决定。
- 如果有更简单的方案，应主动指出；必要时要适度反驳。
- 如果有地方看不清，就先停下；指出哪里不清楚，再提问。

## 2. 简单优先

**只写解决当前问题所需的最少代码，不做任何猜测性扩展。**

- 不要添加用户没要求的功能。
- 不要为一次性代码做抽象。
- 不要加入未被要求的"灵活性"或"可配置性"。
- 不要为不可能发生的情况添加错误处理。
- 如果你写了 200 行，但其实 50 行就够，就重写到更简单。

随时问自己：**"一个资深工程师会觉得这实现过度复杂了吗？"** 如果答案是会，那就继续简化。

## 3. 手术式修改

**只改必须改的地方。只清理由你自己造成的问题。**

在修改现有代码时：
- 不要顺手"优化"相邻代码、注释或格式。
- 不要重构那些并没有坏的部分。
- 尽量匹配现有风格，即使你个人会写成别的样子。
- 如果你发现无关的死代码，可以提一句，但不要直接删。

当你的改动产生了"遗留物"时：
- 删除那些因为你的改动而变成未使用的 import、变量、函数。
- 不要删除原本就存在的死代码，除非用户明确要求。

检验标准：**每一行改动，都应该能直接追溯到用户的请求。**

## 4. 目标驱动执行

**先定义成功标准，再循环推进，直到验证通过。**

把任务改写成可验证的目标：
- "加校验" → "先写出非法输入的测试，再让测试通过"
- "修这个 bug" → "先写出能复现问题的测试，再让测试通过"
- "重构 X" → "确保改动前后测试都通过"

对于多步骤任务，先给出一个简短计划：

```text
1. [步骤] → 验证方式：[检查项]
2. [步骤] → 验证方式：[检查项]
3. [步骤] → 验证方式：[检查项]
```

强成功标准能让你独立迭代；弱成功标准（比如"把它弄好"）只会导致反复澄清。

---

**如果这些规则生效，你应该看到：**
- diff 里的无关改动更少
- 因为过度设计导致的返工更少
- 澄清问题发生在实现之前，而不是出错之后
