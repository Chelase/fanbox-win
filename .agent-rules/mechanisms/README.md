# 机制文档目录（mechanisms/）

本目录存放项目稳定机制文档，描述系统当前已经存在的链路、协议、数据契约、风险边界。

## 判据

写"系统现在如何工作、接口如何协作、字段如何解释"，放本目录。
若是"必须/不要"的执行约束，放 `../rules/`；若是"本期要改成什么样"，放 `../dev-plans/`。

## 约定

- 一个机制主题一个文件，文件名用语义化短横线命名，例如 `auth-flow.md`、`data-pipeline.md`。
- 机制文档描述"系统是什么样"，是开发计划的事实依据；开发计划顶部应反向链接到对应机制文档。
- 机制约束、协议契约、风险条款集中写在机制文档里，开发计划只引用不复制。
- 新增 / 删除机制文档后，同步更新本清单与 `../README.md` §6（双索引防漂移）。
- 每当 Agent 完成分析、梳理、排障复盘、架构理解或项目链路确认任务后，若产出了后续开发或维护可复用的稳定事实，必须新增或更新相应机制文档。
- 机制文档保持简洁，聚焦机制运行链路；禁止写的内容见下方「禁止写的内容」清单。
- 若本目录文档很多，不要全量读取或逐个文件盲找；先读本 README 索引，用关键词定位相关主题，再精读目标文档。
- 当本目录文档数量明显变多（例如超过 12 个文件，或已经需要滚动查找）时，按业务域、子系统或能力主题拆分子目录，例如 `auth/`、`data/`、`api/`。
- 每个子目录必须包含自己的 `README.md`，说明覆盖范围、文档清单、推荐读取顺序和上级索引中的入口。

## 机制文档结构（七段强制）

每份机制文档**必须**包含以下七个段落，缺一不可。顺序固定，标题名可微调但语义不能变：

1. **结论**：一句话说明这条机制解决什么事。
2. **涉及对象**：表格式列出相关文件、接口、表，每项注明角色。格式固定为三列：`层 | 文件/表 | 角色`。
3. **运行链路**：ASCII 图按步骤描述从入口到结果的完整流转。链路图是文档核心——先画图再写文，图覆盖不到的地方用文字补。缩进层级表达调用深度，箭头表达数据流向，方括号表达所在模块。
4. **使用点**：列出当前有哪些地方调用了这条机制（"谁、在什么时机、调了什么"），用于评估改动影响面。
5. **修改点**：表格式给出「要做什么 → 改哪里 → 怎么改」，让后续开发者无需全文重读即可定位修改文件和操作顺序。格式固定为三列：`要做什么 | 改哪里 | 怎么改`。
6. **关键约束**：只写后续开发必须注意的不变量、兼容边界和易错点。不超过 6 条，超过说明机制梳理不够精或需要拆分文档。
7. **维护方式**：给出新增、排查、验证的具体操作方法（"调哪个接口看什么字段""按什么顺序改哪些文件"）。

末尾写 `> 更新时间：YYYY-MM-DD`。

### 禁止写的内容

以下内容**一律不得**出现在机制文档中，出现即违规：

- **大段代码示例**：代码应读源文件，机制文档最多保留 1-2 行关键调用签名。
- **方案论证与决策树**：属开发计划（`dev-plans/`），不属于"系统是什么样"。
- **环境配置表、DDL 语句**：属 README 或运维文档。唯一例外：存量数据一次性补偿 SQL 可在维护方式中引用。
- **长篇背景说明**：属架构总览（`docs/`），机制文档从"系统现在的运行方式"开始写。
- **超过 150 行**：超过说明梳理不精或范围过大，需要提炼或拆分为多篇。
